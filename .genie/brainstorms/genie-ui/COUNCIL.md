# Council Report: Deliberate for a tired, confused owner who asked the council to decide FOR them (they will ratify, not re-derive). The decision: the genie UI wish's architecture, specifically whether to commit to the "two-faces" model. Full context lives in /home/namastex/prod/ryzen-ai-station/.genie/brainstorms/agent-conductor/DRAFT.md (read it — it has the recovered plan, research digest in sibling RESEARCH.md, and draft acceptance criteria at WRS 80).

The settled facts (do not re-litigate): UI = fork of syv-ai/dash pattern (renders THE agent as real PTY panes, zero UI abstraction — owner's hard value); left menu = genie wishes instead of projects; each wish hires agents (one Claude Code, one Codex, one Hermes profile, rlmx); tabs/splits swap agents within a wish context; a wish-scoped GROUP CHAT drawer where hired agents + human converse; ACP is the inter-agent bus; ONE wish for all of it; home = the genie repo (automagik-dev/genie, old v4 TUI already removed from tree); an A/B build-off (fork-dash vs fresh shell) is running separately and will decide the shell foundation empirically.

The open question the council must settle: HOW does an agent participate in both its terminal pane and the group chat? Candidate models: (A) chat-only agents whose "terminal" renders the ACP stream — violates the owner's no-abstraction value; (B) "two faces, one workspace" — each hired agent gets an interactive PTY face AND a sibling ACP session, both bound to the wish worktree, shared truth = wish artifacts in git + per-harness session-memory bridging (Hermes shares state.db across surfaces; Claude Code JSONL transcripts replayable by both faces; Codex unknown; rlmx ours); (C) inject chat messages into the PTY as keystrokes and scrape replies from the terminal — purest but parsing-fragile; (D) any hybrid or alternative a lens argues is genuinely better (e.g. per-harness: C-style injection where a harness has a clean prompt-input mode, B elsewhere; or chat-face-only agents with NO terminal for some roles). Secondary questions if the council has strong signal: whether the group chat should deliver @-mentions only vs all-messages-to-all-agents (cost/noise), and whether session-memory bridging (two faces = one memory) should be a v1 requirement or a per-harness best-effort.

Deliberate across lenses (architecture/simplicity, product/UX-fidelity to the owner's values, operational cost/token burn, risk/failure modes), argue genuinely, then synthesize ONE recommendation with a crisp rationale the owner can ratify in one sentence, plus what it changes in the draft acceptance criteria. The owner's state matters: they need a confident, decided answer, not a menu.

## Executive Summary
The council converged 4-0, across two rounds and all four lenses, on Model B — two faces, one workspace — with one decisive correction to the draft: the coherence contract is the shared wish worktree and git artifacts, NOT shared session memory. Session bridging is demoted to per-harness best-effort (Hermes demonstrated via state.db; others exempt), the chat face is non-mutating in v1, spawns lazily on first @-mention, and delivery is @-mention-only. The one-sentence ratification for the owner: "Each hired agent is one real terminal plus one lazily-spawned, read-only ACP chat face in the same wish worktree; git artifacts are the shared memory, @-mentions route the chat, and Hermes-style session bridging is a demonstrated bonus, never a gate."

## Consensus
- Model B wins outright (4-0): Model A deletes the PTY the owner forked dash to keep; Model C is a bespoke keystroke-injector/output-scraper per harness — 'a permanent maintenance annuity disguised as minimalism' — with unnameable failure modes; per-harness D-hybrids are a matrix re-derived on every harness change. B adds exactly one concept and all four ACP adapters already exist.
- The identity knot dissolves by reframing: shared truth = wish worktree + git artifacts (genie's existing model), not cross-face session identity. An AC pinned to four vendors' undocumented session internals has maximal change amplification; an AC pinned to 'both faces cd into the same worktree' depends only on things the owner controls.
- Session-memory bridging is per-harness best-effort, never a v1 gate: demonstrate on Hermes (state.db is a vendor contract, not an inference), CC JSONL-resume as stretch, Codex/rlmx explicitly exempt. A gate passable on two of four harnesses 'is not a gate, it's a wish' (questioner).
- @-mention-only delivery in v1, firmly: broadcast fans every chat line into N inference calls, invites reply cascades, and makes room behavior emergent and unteachable. It also matches the conductor framing — 'who speaks NEXT in the room.' Broadcast is a later routing policy on the same bus, and docs must state @-mention-only as the designed contract, not a temporary limitation.
- Lazy face spawn: hiring creates a roster entry only; the PTY face spawns on tab-open, the ACP face on first @-mention (seeded with wish context + room transcript per architecture's spawn contract). This delivers 'chat-only role' agents as a usage pattern with zero new machinery, dissolving Model D's special agent type.
- The chat face is non-mutating in v1 (adopted by all four after questioner's write-collision hazard): two write-capable processes of one agent in one worktree is an unserialized concurrent-writer design and a guaranteed week-one race. Contract: the terminal is where the agent WORKS, the chat is where it TALKS, git is what both read.
- The chat backend is a separable module (one ACP ClientSideConnection per hired agent; interface = deliver message, stream reply) with no PTY-layer imports — structurally identical to the conductor's future hub, making wish one the literal substrate of wish two. Also settles B-over-C: auto-advance cannot be built on scraped terminal output.
- Delivery/spawn failures surface as named, greppable chat-drawer events ('@codex could not start: codex-acp exited (code 127); check PATH'), never silence — otherwise lazy spawn degrades into the 'room went quiet' failure Model C was rejected for.

## Tensions
- Promotion toggle vs no toggle (2-2, the only genuine split): questioner and dx-docs want 'non-mutating by default, write access an explicit promotion'; simplifier and architecture cut the promotion entirely — it adds a modal state per chat face, a promotion UI, and a per-harness permission-mapping matrix (codex-acp has real read-only mode; CC/Hermes need bespoke constraint translations; rlmx is unbuilt). The synthesis sides with no-toggle for v1 on the same deferral logic all four applied to broadcast; the promotion view is preserved in Dissent.
- Capability badge granularity: dx-docs proposed a three-tier badge ('shared memory' / 'transcript-linked' / 'separate context'); simplifier called the taxonomy 'the Model D matrix sneaking back as chrome' and cut it to one default statement plus a single 'shared memory' badge for Hermes; questioner flagged that 'transcript-linked' for CC overclaims (plausible, not demonstrated) and added a write-capability dimension; architecture required badges derive from machine-readable adapter capability declarations, not hand-maintained copy. Convergent resolution: badges yes, minimal set, driven by a checked-in per-harness capability table (needed anyway since lazy spawn means no live session to probe at hire time).
- Procedural: architecture and dx-docs minted conflicting AC6s (chat-backend module boundary vs error surfacing). Questioner's resolution — keep both, number separately — is adopted below as AC6 and AC7.

## Evolution
Round 1 produced independent 4-0 convergence on the core (B, git-as-truth, @-mention-only, bridging demoted) — the owner should read that independence as strong signal. Round 2 was integration, not reversal. Three things moved: (1) questioner's write-collision hazard — absent from the draft (the words collision/race appear nowhere in DRAFT.md) — was unanimously adopted as the round's strongest point and hardened into an AC-level invariant; architecture conceded it as 'the one load-bearing failure mode my own position missed.' (2) Lazy face spawn spread from questioner to all four, then got disciplined: architecture added the spawn contract (seed wish context + room transcript, or the first @-mention gets an amnesiac reply), and dx-docs added that spawn failure becomes the new error surface. (3) Guards were simplified by subtraction: simplifier and architecture stripped questioner's promotion toggle ('the race is defined out of existence rather than guarded'), and dx-docs's badge taxonomy was pared to one honest default plus one demonstrated exception. Net effect: the consensus draft AC text ('readable/writable by both faces') was caught re-importing the race and corrected before ratification.

## Dissent
On the write-promotion toggle (council split 2-2; synthesis sided with no-toggle for v1):

Questioner (Round 2): "The consensus AC text is re-importing the race I flagged: architecture's AC4a says wish artifacts are 'readable/WRITABLE by both' faces — ratifying two write-capable processes of the same agent in one worktree (worst for CC, where 'shared memory' means two live processes appending the same JSONL). Required clause: chat face non-mutating by default, write access an explicit promotion."

dx-docs (Round 2): "I adopt the non-mutating chat face by default with explicit promotion (questioner), and I extend my badge proposal to carry it: each chat face shows both its memory state AND its write state, so the work/talk division is visible, not tribal knowledge." And from Round 1's rationale: "read-only-by-default with explicit promotion turns an undocumentable race into a documented mode switch."

On the badge taxonomy (dx-docs's fuller proposal, pared back by the synthesis): dx-docs (Round 1): "every hired agent's chat face carries a visible badge — 'shared memory' (Hermes), 'transcript-linked' (CC), 'separate context' (Codex until proven) — reference documentation delivered at the moment of need, which is what DX is."
