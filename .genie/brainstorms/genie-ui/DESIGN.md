## Problem

The user's coding agents (Hermes profiles, Claude Code as Fable and Opus, Codex, rlmx) live in separate harnesses with no shared environment; wiring any of them into an ACP client is high-friction (especially over network), and the native-mac ACP clients that could host them abstract each agent into a thread UI — the exact opposite of the user's hard value, which is to *watch the real agent's own terminal*. The result is that cross-harness work (a wish's author, reviewer, executor, and gate) is manual copy-paste between disconnected panes, and there is no surface that renders genie's own state (wishes, board, worktrees) next to the agents doing the work.

The reframe that dissolves the "which client" question: **the planned genie UI IS the environment.** Don't adopt a host — render THE agent (the `syv-ai/dash` pattern: a real PTY pane per agent, zero rendering abstraction) and layer genie's wish/board state and a wish-scoped inter-agent chat on top. This matters because it is the only shape that keeps the user's no-abstraction value while still giving the agents a machine-readable channel to collaborate.

## Scope

### IN

- **The fleet floor (PTY viewing channel).** A browser-served UI whose panes each run THE real agent surface (`claude`, `hermes -p <profile> --tui`, `codex`, rlmx/pi TUI) over a local PTY or plain `ssh -t` — PTY-faithful, no rendering abstraction. Tabs swap agents within a wish; 2+ panes split horizontally; per-pane spawn/kill/restart; screen replay on (re)attach via a compact snapshot engine. Ported from the A/B "fresh" substrate and hardened with salvaged Lane-A machinery (below).
- **The genie lane (left menu).** The left menu lists genie **wishes** (from the repo `.genie/wishes` markdown + `.genie/genie.db` board state), replacing dash's Projects. Selecting a wish opens its worktree-bound context. A **hire roster** binds fleet members (CC / Codex / Hermes profile / rlmx) to a wish; hiring is a roster entry only. Worktree binding is **per roster entry** and **reuses** `genie launch`'s existing per-group worktree for the ready group that agent is on (one worktree per ready group) — it never mints a parallel wish-level worktree (see the worktree wall for the exact mapping).
- **The wish group chat (ACP control channel).** A wish-scoped group-chat drawer where hired agents + the human converse. Each hired agent gets one **lazily-spawned, read-only ACP chat face** (one ACP client connection per agent) in **that agent's own worktree** — the same per-group worktree its terminal face runs in (the coherence contract is per-agent, not a single shared wish path; the shared truth across agents is the `.genie` git artifacts — see the worktree wall). `@agent` delivers the message + room transcript to that agent's chat face; replies stream back as chat messages. Delivery is **@-mention-only** as the designed contract. The chat backend is a **separable module with no PTY-layer imports** — the literal substrate the future conductor wish routes on.
- **Per-harness capability table**, checked into the repo, driving minimal chat-face badges (e.g. "shared memory" for Hermes) at hire time.
- **Old-UI residue check + docs.** Confirm no LIVE code references to the removed `packages/genie-app` / `src/tui/` era remain (v5-demolition already scrubbed them; historical `CHANGELOG.md` / completed-wish records are preserved, not scrubbed); document the two-faces contract and the two channels.
- **New code lands as a package** (`packages/genie-ui`): a small bare-`ws` PTY server entry plus the browser client, following the repo's current package conventions (single-file bun bundle for the CLI stays untouched; the UI server is its own entry).

### OUT

- **Conductor auto-pilot / auto-advance routing** — the policy layer that makes the room advance the pipeline (Fable wish → Hermes review → Codex execute → Opus fix → Fable gate) automatically instead of by human @-mentions. This is a **follow-up wish** built as a routing-policy layer on top of this wish's chat-backend module; explicitly not in v1. (DRAFT: "conductor auto-pilot = follow-up wish.")
- **Chat-face write-promotion toggle** — the ability to promote a chat face from read-only to worktree-mutating. The council split 2-2 on this (Dissent, COUNCIL.md); the synthesis defers it. The chat face is non-mutating in v1; write access is a **documented later extension point**, not a v1 mode. Preserving it as OUT is what keeps the two-writers-one-worktree race defined out of existence (see AC4a / Decision D5).
- **Broadcast delivery / all-messages-to-all-agents routing** — deferred as a later routing policy on the same bus. Broadcast fans every chat line into N inference calls, invites reply cascades, and makes room behavior emergent and unteachable; @-mention-only is the *designed* contract, not a temporary limitation.
- **Electron / Tauri desktop packaging** — dash is Electron; genie-ui is **browser-served from the box** (works from mac + phone immediately, matching the user's multi-device habit). Desktop packaging is out; no Electron/Tauri dependency enters the tree.
- **Session-identity coupling as the coherence contract** — an AC pinned to four vendors' undocumented session internals is explicitly rejected. Coherence = shared worktree + git artifacts. Session bridging is best-effort (AC4b), never a gate.
- **Model A (chat renders the ACP stream) and Model C (keystroke injection + output scraping)** — both rejected by the council (Model A deletes the PTY the user forked dash to keep; Model C is an unnameable-failure-mode maintenance annuity on which auto-advance cannot be built).
- **Newio / any external host** as the environment — the genie UI *is* the environment. Newio remains a possible later *social* surface, not this wish.
- **Station strip / structure sidecar / QR-remote reach** (metal-river telemetry, rlmx recursion tree, phone-grab) — pillars 4-6 of the reconstructed vision. Cheap later additions on data planes that already exist; not v1 scope.

## Approach

**Two channels, not competing** — the design's spine (DRAFT "Reframe"):

1. **PTY viewing channel (dash pattern).** Each fleet member is a pane running its real surface, locally or over plain `ssh -t` (PTY-over-SSH is native — the network pain disappears for the *viewing* channel). Zero abstraction: you watch the actual agent. This is the thing the user loves and forked dash to keep.
2. **ACP control channel (the conductor substrate).** genie's role graph needs machine-readable events, not pixels. Each hired agent has a sibling **read-only ACP chat face** (a `ClientSideConnection` to the harness's existing ACP adapter — three ship today: `@agentclientprotocol/claude-agent-acp`, `@agentclientprotocol/codex-acp`, `hermes acp`; `rlmx acp` is **in flight** (wish `rlmx-acp-adapter`, RESEARCH §R2). rlmx is exempted by AC4b and an unavailable adapter surfaces as a D9 fail-loud event, so v1 does not block on it). A chat message `@agent` becomes an ACP `session/prompt` (+ room context); replies stream back as chat messages. Artifacts stay in git.

**The one hard knot — one agent, two faces — and its ratified resolution (Model B, corrected).** The PTY tab (the agent's real TUI) and the chat-drawer participation (structured in/out) cannot be the same OS process: TUIs speak pixels, the chat needs text/events. The council settled the identity knot by *reframing it away*: the two faces of one agent do not need to share a session; they share **that agent's worktree + git artifacts** (genie's existing git-as-truth model). "Wish worktree" in the ratified contract below resolves per-agent to the `genie launch` per-group worktree that agent is on — not a single wish-wide path; the worktree wall pins the exact mapping. The ratified one-sentence contract, verbatim:

> **"Each hired agent is one real terminal plus one lazily-spawned, read-only ACP chat face in the same wish worktree; git artifacts are the shared memory, @-mentions route the chat, and Hermes-style session bridging is a demonstrated bonus, never a gate."**

The council's corrections to the pre-council draft, verbatim where load-bearing (DRAFT "Corrections to the draft"; all P0 unless noted):

> coherence contract = worktree + git artifacts, NOT session identity; chat face NON-MUTATING in v1 (terminal works, chat talks, git is what both read — kills the two-writers-one-JSONL race); @-mention-only delivery as the *designed* contract; lazy spawn (PTY on tab-open, ACP face on first @-mention seeded with wish context + room transcript) — which makes "chat-only role agents" free (P1); chat backend = separable module (1 ClientSideConnection per hired agent, no PTY imports) = the literal conductor substrate for wish two; delivery/spawn failures are named chat-drawer events, never silence.

**Alternatives considered and why they lost** (COUNCIL.md Consensus): *Model A* — chat-only agents whose terminal renders the ACP stream — deletes the PTY that is the user's whole point (rejected). *Model C* — inject chat as PTY keystrokes and scrape replies — "a permanent maintenance annuity disguised as minimalism," with unnameable failure modes, and auto-advance cannot be built on scraped terminal output (rejected). *Model D* per-harness hybrids — a matrix re-derived on every harness change (rejected). *Session-identity coupling* — maximal change amplification against four vendors' undocumented internals (rejected as the coherence contract; kept only as best-effort AC4b). *Adopting a host (Zed/Newio/Tidewave/native-mac)* — all abstract the agent into a thread UI, the opposite of the value (rejected: the UI IS the environment).

**Shell foundation (A/B verdict, ratified).** Both prototypes at `~/prod/genie-ui-ab/{dash-fork,fresh}` passed all 7 acceptance items (including the `ssh -t` pane). Lane **B "fresh" wins** as the substrate — a bare-node `ws` server (~250 lines vanilla TS), `node-pty` + `@xterm/xterm` 5.5 + fit, fleet config, `idle/running/exited` status events — **ported into `packages/genie-ui`**. ("Zero build step" describes the **server** (bare `ws`, run directly under bun). The prototype's **client**, however, imported `@xterm` as a bare specifier and leaned on **Vite** to transpile/serve it (`"client": "vite"`) — a build step the genie repo (bun-only, no Vite) should not adopt. G1 replaces Vite with a **bun-native serve path**: `Bun.serve` transpiles the client TS on the fly and resolves `@xterm` via an import map (or a minimal `Bun.build` of the single client entry). This is a G1 decision to pin, not an implicit "no tooling at all.") Salvaged from Lane A: `server/reused/TerminalMirror.ts` (headless-`xterm` + `SerializeAddon` compact snapshot engine — **replaces B's 256 KB raw-byte ring replay**, B's key liability), the `Utf8Base64` + `FitScheduler` utilities, and the grid/maximize/splits layout **concept** (reference only, reimplemented vanilla — Lane A's other liability was single-active-tab layout). Everything honors repo conventions: **bun** runtime, `bun:test` colocated tests, biome (single quotes, 2-space, 120 width), commitlint conventional, cognitive-complexity budget 25.

## Architecture (modules · interfaces · isolation)

New package `packages/genie-ui`. Two entries: a **PTY/ws server** (bare `ws`, ported from fresh) and the **browser client** (served from the box). The chat backend is a **third, independently importable module** with a hard isolation wall. Ported/salvaged module names below map onto the A/B prototypes so the executor can trace provenance.

### Isolation contract (Design for Isolation — the invariants a reviewer greps for)

- **The PTY wall.** `node-pty` is imported in exactly one module (`pty-session`). Everything above it (transport, client, genie-lane, chat-backend) sees only the `PtySessionManager` interface + event shape. Swapping a fleet member's real CLI (fable/hermes/codex/rlmx) touches only `fleet-config`.
- **The chat wall (load-bearing).** `chat-backend` imports **nothing** from `pty-session` / `TerminalMirror` / `transport` / `client`. Its interface is `deliver message / stream reply`. This is structurally identical to the conductor's future hub (a `ClientSideConnection`-per-child router), so wish one is the literal substrate of wish two — verified by a greppable "no PTY imports" test, not an aspiration.
- **The state wall.** genie state is read-only in this UI. `genie-lane` reads `.genie/wishes` (git) + `.genie/genie.db` (`bun:sqlite`, read-only open, degrade-to-empty when absent — the `genie mcp` precedent) or reuses the `genie mcp` read layer. **No new write path** into genie.db races the CLI/skills.
- **The worktree wall (coherence contract) — the exact wish↔worktree mapping.** The contract binds **per hired agent, not per wish**: the two faces *of one agent* (its terminal + its read-only chat face) `cd` into the **same worktree**, and that worktree **is** the `genie launch` per-group worktree for the ready group that agent is hired onto — reused, never a parallel wish-level worktree minted alongside it (that is how it *composes with, does not duplicate*, `genie launch`; `warp-launch.ts` = one worktree per ready group). So `worktreeFor` resolves against a **roster entry** (agent + its group), not a bare wish: a wish with N ready groups fans out into N group worktrees, and hired agents live in whichever group worktree they are bound to — there is no single "wish worktree" path. Two coherence layers make this consistent:
  1. **Per-agent (filesystem):** an agent's terminal face mutates, its chat face reads, both in that agent's one group worktree. The terminal face is the sole mutator; the chat face is read-only.
  2. **Wish-level (git):** across all group worktrees the `.genie` wish artifacts are the shared truth — git-tracked `.genie/wishes/<slug>/` (main-repo tree) plus the worktree-shared `.genie/genie.db` (resolved via `git rev-parse --git-common-dir`, one DB across every worktree). This shared `.genie`, not a single filesystem path, is what makes the group chat *wish-scoped* even when its participants sit in different group worktrees.
  Before its group is launched, a roster entry is **unbound** (a roster row only); its faces resolve a worktree at spawn time once `genie launch` has created the group worktree. This depends only on things the user controls (worktree paths + git artifacts), not on vendor session internals.
- **Server ↔ client.** The browser talks to the server exclusively over the `transport` `ws` protocol; every client (tab/phone) renders the whole fleet, kept in sync by a single broadcast.

## Decisions

| # | Decision | Rationale |
|---|----------|-----------|
| D1 | **The genie UI is the environment; render THE agent (PTY-faithful), no host adoption.** | The user's hard value is watching the real agent; every surveyed host (Zed/Newio/Tidewave/native-mac) abstracts the agent into a thread UI (the opposite). Two channels — PTY viewing + ACP control — instead of one abstracted thread. |
| D2 | **Ratify Model B, corrected: two faces, one workspace; coherence = worktree + git artifacts, NOT session identity.** | Unanimous 4-0 across two rounds. The smallest design that tells no lies, uses only published vendor contracts (all four ACP adapters ship today), and extends genie's git-as-truth model to the UI rather than inventing one. An AC pinned to "both faces `cd` into the same worktree" depends only on what the user controls. |
| D3 | **Left menu = genie wishes; per-wish hired agents as tabs + horizontal splits.** | The wish is the organizing layer with coding agents on it (recovered plan). Same wish context, multiple simultaneous agent faces. |
| D4 | **@-mention-only delivery, as the *designed* contract (not a temporary limit); undelivered chat is visible history, not implicit context.** | Broadcast = a token/noise multiplier with emergent, undocumentable room behavior; @-mention is the primitive the conductor wish drives ("who speaks next"), so it forecloses nothing. Broadcast is a later policy on the same bus. |
| D5 | **Chat face is non-mutating in v1 (read-only ACP face); only the terminal face mutates the worktree.** | Two write-capable processes of one agent in one worktree is an unserialized concurrent-writer design — a guaranteed week-one race (worst for CC: two live processes appending one JSONL). "shared memory" badge only where demonstrated (Hermes). |
| D11 | **Shell foundation = fresh substrate ported + Lane-A salvage (TerminalMirror snapshot, Utf8Base64/FitScheduler, grid/splits concept).** | A/B verdict: both passed all 7 items; fresh's bare-`ws`/zero-build substrate wins; its two liabilities (256 KB ring replay, single-active-tab layout) are fixed exactly by the salvaged TerminalMirror + vanilla grid/splits. |
| D12 | **Browser-served from the box; no Electron/Tauri.** | Works from mac + phone immediately (multi-device habit); no desktop-packaging dependency enters the tree. Electron packaging is OUT. |
| D13 | **New code = `packages/genie-ui` (separate UI-server entry); read genie state read-only; compose with `genie launch` worktrees.** | Follows the repo's package pattern (the CLI's single-file bun bundle stays untouched); no second write path into genie.db; worktree binding composes with, not duplicates, existing per-group worktrees. |
